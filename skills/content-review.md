# Content Review

吸收自 `ai_tribunal_design/content_reviewer.md`。保留其“逐条审查”的优点，但避免把冗长示例直接压进各司。

## 何时使用

- 叙事司编写世界观、角色、人设与文本时
- 玩法司/系统司撰写规则说明时
- 门下省检查方案是否“可读、可做、无歧义”时

## 核心关注点

- 文案是否通顺、无歧义、符合场景语气
- 规则描述是否前后一致，是否存在文字与公式冲突
- 参数是否完整，包括默认值、范围、单位、边界情况
- 核心术语是否统一，是否存在同一概念多种叫法

## 快速检查表

- [ ] 是否有关键参数缺失
- [ ] 概率、百分比、枚举是否自洽
- [ ] 同一机制在不同段落是否自相矛盾
- [ ] 术语是否统一且首次出现可理解
- [ ] 读者能否直接据此继续实现或评审

## 常用分类

- `规则矛盾`
- `参数缺失`
- `术语不一致`
- `歧义表述`
- `完整性缺失`
- `格式规范问题`

## 严重级别参考

- `CRITICAL`
  - 关键规则互相冲突，按文档无法唯一实现
  - 核心参数缺失导致研发、数值或测试无法继续
- `HIGH`
  - 同一机制在不同章节给出不同结论，且会影响玩法或经济结果
  - 单位、范围、枚举缺失，易造成错误实现
- `MEDIUM`
  - 术语未统一、文案存在歧义，但通过补定义可快速修正
  - 描述顺序混乱，影响评审效率
- `LOW`
  - 格式、标题、排版或局部措辞不稳
- `INFO`
  - 建议增加术语表、参数表、示例流程图

## 示例发现

- `CR-EX1`
  - 分类：`规则矛盾`
  - 场景：前文写“每日可挑战 3 次”，后文又写“体力足够可无限挑战”
  - 为什么严重：实现规则无法唯一确定，测试口径也会分裂
- `CR-EX2`
  - 分类：`参数缺失`
  - 场景：奖励掉落写“低概率获得”，但没有给区间或配置表入口
  - 为什么严重：研发无法落表，测试无法验收
- `CR-EX3`
  - 分类：`术语不一致`
  - 场景：同一资源在数值表里叫“荣誉勋章”，正文里叫“战区徽记”
  - 为什么严重：跨团队沟通会不断打架，容易接错表

## 不要这样报

- 不要把纯文风偏好、修辞喜好报成结构性问题
- 不要只说“看不懂”，必须指出哪一句、哪一段、缺了什么信息
- 不要把实现细节缺失和业务决策未定混为一谈
- 没有跨段冲突或实际歧义时，不要把正常简写硬报成问题
