# Mobile Control Selection

把“这类玩法应该怎么操作”从直觉判断变成轻量可复用的选择规则。

## 何时使用

- 玩法方案需要确定移动、攻击、交互的输入方式
- 要做 H5 手游原型，但还没决定是摇杆、点触、按钮还是自动镜头
- 需要把玩法特征转换成适合移动端的交互模型

## 判断顺序

1. 先看玩法是否真的需要连续移动
2. 再看是否需要玩家手动控制镜头
3. 再看战斗 / 技能是否需要并行输入
4. 最后判断是否要混合多种输入方式

## 典型映射

- 动作、探索、地牢、生存、移动射击
  - 默认：`摇杆 + 按钮`
- 建造、经营、拖拽、放置
  - 默认：`点击 / 拖拽`
- 卡牌、回合、三消、视觉小说
  - 默认：`点击 / 按钮`
- 只有攻击技能、没有持续移动
  - 默认：`按钮`
- 需要空间移动但不适合自由镜头
  - 默认：`摇杆 + 自动镜头`

## 冲突修正规则

- 若玩法描述同时要求“自由镜头 + 摇杆 + 竖屏单手高频操作”，默认优先保留 `摇杆`，镜头改为 `自动镜头`
- 若玩法不依赖位置移动，只依赖目标选择或资源操作，默认移除摇杆
- 若攻击与移动都要高频发生，应保证两者能并行，不要让同一触控区互相抢输入
- 若某种输入方式只是“看起来像手游”，但不服务核心玩法，应删除

## 输出建议

输出时至少给出：

- 选择的操作方式
- 为什么选它
- 为什么没选其他方式
- 可能的输入冲突与修正方案

## 最小输出模板

```markdown
操作方式：
- [摇杆 / 点击 / 按钮 / 自动镜头 / 混合]

选择理由：
- [对应玩法意图]

冲突处理：
- [移动、攻击、镜头、UI 是否冲突]

不采用的方式：
- [为什么不选其他方案]
```

## 常见错误

- 把“手游”直接等同于“必须摇杆”
- 没有移动需求却硬加移动输入
- 同时保留过多输入方式，导致学习成本和冲突一起上升
- 镜头、移动、攻击都要玩家手动管，超出竖屏原型承载能力
