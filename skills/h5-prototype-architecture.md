# H5 Prototype Architecture

用于把“先做一个能跑的 H5 手游原型”落成清晰、可迭代的轻量工程结构。

## 何时使用

- 需要规划原生 H5 原型目录结构
- 需要约束配置、输入、资源加载、主循环的职责边界
- 需要避免原型阶段把逻辑全堆进一个入口文件

## 设计顺序

1. 先定义最小运行骨架
2. 再拆分配置、工具、加载、输入、主循环
3. 再为新增玩法功能追加模块
4. 最后补资源预算和性能观察点

## 最小模块建议

- `index.html`
- `css/style.css`
- `js/config.js`
- `js/utils.js`
- `js/loader.js`
- `js/input.js`
- `js/main.js`

新增玩法或系统时，再按需新增：

- `js/ui-*.js`
- `js/scene-*.js`
- `js/system-*.js`
- `js/entity-*.js`

## 模块职责

- `config.js`
  - 放频繁调整的数据、尺寸、速度、颜色、文本、资源路径
- `utils.js`
  - 放纯工具函数
- `loader.js`
  - 放资源加载与失败 fallback
- `input.js`
  - 放触屏 / 鼠标输入映射
- `main.js`
  - 放启动、主循环、状态分发，不堆全部业务细节

## 加载与修改原则

- 脚本按依赖顺序加载，避免隐式前置依赖
- 新功能优先新增模块，不默认重写 `main.js`
- 改数值先改配置
- 改输入先改输入层
- 改 UI 先改 UI 模块与配置

## 默认原型约束

- 优先保证 `file://` 或轻量本地环境下可快速验证
- 优先本地资源，避免原型阶段依赖外网
- 优先竖屏移动端布局；若需求明确要求横屏或宽视野，再另行说明
- 资源加载失败时，应给出占位或降级方案，而不是直接黑屏

## 输出建议

输出 H5 原型结构方案时，优先包含：

- 最小目录
- 核心模块职责
- 配置归属
- 新功能应该落在哪类文件
- 当前不做什么

## 常见错误

- 把全部逻辑、输入、状态、UI 都堆进 `main.js`
- 把频繁调整的数值散落在多个模块里
- 为了一个原型直接引入过重目录层级
- 一修改功能就整体重写入口和样式文件
