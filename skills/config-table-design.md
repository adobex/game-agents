# Config Table Design

把策划方案中的可调数据抽取成配置表，避免硬编码和字段混乱。

## 何时使用

- 方案需要落配置表
- 需要定义字段名、类型、示例值
- 需要设计表间关系、外键或 JSON 字段

## 设计顺序

1. 先列“哪些数据会频繁调整”
2. 再分表：基础配置 / 奖励 / 消耗 / 进度 / 文本
3. 再定字段：名称、类型、含义、默认值
4. 最后补关系：主键、外键、映射表

## 核心规则

- 字段名优先 `snake_case`
- 主键 / 外键统一用 `xxx_id`
- 枚举值必须写清含义
- 复杂结构优先 JSON，不用自定义拼接字符串
- 说明 `0`、`-1`、空字符串分别代表什么
- 所有配置表至少包含用途说明和示例值

## 常见表类型

- 基础功能配置：开启条件、周期、冷却、开关
- 奖励配置：奖励内容、权重、保底、档位
- 消耗配置：货币、体力、道具、次数限制
- 进度配置：等级、阶段、目标数量、里程碑
- 文本配置：提示、描述、多语言文本

## 输出模板

```markdown
表名：XxxConfig
用途：[这张表配置什么]

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| xxx_id | int | 主键 | 1001 |
| name | string | 名称 | "新手礼包" |
| is_active | bool | 是否启用 | true |
```

## 关系设计提醒

- 一对多：在从表放外键
- 多对多：明确中间映射表
- 奖励、掉落、活动参数等常需要独立表而不是塞进单表

## 常见错误

- 同义字段反复换名
- 魔法数字没有枚举说明
- JSON 字段缺结构示例
- 奖励、消耗、活跃度等职责混在一张表里
