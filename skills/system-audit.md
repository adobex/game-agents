# System Audit

吸收自 `ai_tribunal_design/system_architect.md`，但只保留项目内高频复用的审查骨架。

## 何时使用

- 系统司设计外围系统时
- 战斗司设计战斗/技能/AI 关系时
- 关卡司涉及事件驱动与状态流转时
- 门下省需要检查“结构是否站得住”时

## 核心关注点

- 模块边界是否清晰，是否出现“上帝系统”
- 系统间是否存在循环依赖、越层调用、数据竞写
- 是否支持数据驱动扩展，而不是硬编码堆逻辑
- 核心数据的归属权、生命周期、同步路径是否明确

## 快速检查表

- [ ] 输入/输出接口是否明确
- [ ] 是否存在 A 依赖 B、B 又依赖 A
- [ ] 表现层是否越权直接改核心数据
- [ ] 新增内容接入是否需要修改核心逻辑
- [ ] 是否缺少必要状态机或事件流说明

## 常用分类

- `循环依赖`
- `层级违规`
- `上帝系统`
- `数据归属冲突`
- `扩展性缺陷`
- `状态机缺陷`

## 严重级别参考

- `CRITICAL`
  - 核心数据有多个直接写入口，无法保证一致性或回滚
  - 核心状态流转缺失，导致流程可能卡死、重复结算或越权推进
- `HIGH`
  - 明确存在循环依赖、越层调用、表现层直写核心数据
  - 新内容接入必须改动核心逻辑，扩展成本明显不可控
- `MEDIUM`
  - 接口边界大体存在，但事件流、状态转换或归属说明不完整
  - 某些模块职责偏大，但尚未形成致命耦合
- `LOW`
  - 命名、分层表述、文档结构不够清晰
- `INFO`
  - 可选的解耦建议或后续重构方向

## 示例发现

- `SA-EX1`
  - 分类：`数据归属冲突`
  - 场景：背包系统和战斗结算都能直接修改装备耐久
  - 为什么严重：后续容易出现耐久回滚失败、重复扣减、日志对不齐
- `SA-EX2`
  - 分类：`状态机缺陷`
  - 场景：副本从“匹配中”可直接跳到“结算完成”，缺少“载入 / 战斗中 / 结算中”
  - 为什么严重：边界状态无定义，断线和重连无法落位
- `SA-EX3`
  - 分类：`扩展性缺陷`
  - 场景：每新增一种活动类型都要改主循环里的硬编码判断
  - 为什么严重：功能越做越慢，且高风险改动会污染旧功能

## 不要这样报

- 不要只写“架构不优雅”或“耦合偏高”，必须指出哪个模块、什么依赖、会造成什么后果
- 不要把“未来也许能更解耦”的优化建议报成阻塞缺陷
- 不要重复报同一根因的多个表象，应优先归并为一条主问题
- 没有证据时，不要直接判 `CRITICAL`
