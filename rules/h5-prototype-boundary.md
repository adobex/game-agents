# H5 Prototype Boundary

用于约束 `game-agents` 在输出“可试玩 H5 手游原型”相关方案时的默认边界。它吸收 `参考2/H5 手游开发专家.md` 中最有复用价值的部分，但保持为轻量项目规则，而不是整份超级规范。

## 适用场景

- 需要输出原生 H5 手游原型方案
- 需要把玩法快速验证成可本地运行的交互原型
- 需要讨论 H5 原型的工程边界、性能预算、资源约束

## 默认边界

- 默认目标是“快速验证玩法”的 H5 原型，而不是完整商业上线包
- 默认优先 `HTML + Canvas + 原生 JS + CSS` 的轻量原型路线
- 默认优先本地资源与本地运行，不引入构建工具、远程依赖或复杂工程链
- 默认适配移动端竖屏体验；若需求另有明确约束，可被上游旨意覆盖

## 原型优先级

输出 H5 原型方案时，优先保证以下顺序：

1. 玩法可验证
2. 输入方式合理
3. 工程结构可迭代
4. 资源与性能不过载
5. 视觉表现再逐步增强

## 轻量工程原则

- 先给最小可运行骨架，不一次性铺满全部系统
- 配置、输入、主循环、资源加载尽量分离
- 频繁调整的数据优先配置化，避免硬编码散落
- 新增功能优先新增模块，不默认重写主入口

## 默认性能 / 体积预算

以下是原型阶段的建议预算，不是不可覆盖的硬门槛：

- 目标帧率：`60 FPS`
- 若持续低于 `45 FPS`，应优先回头做性能收缩
- 初版资源体积优先控制在轻量范围，避免为验证玩法引入超大素材
- 高成本特效、复杂阴影、重 3D 结构默认后置

## 交互边界

- 是否使用摇杆 / 点击 / 按钮 / 自动镜头，不应拍脑袋决定
- 默认调用 `skills/mobile-control-selection.md` 做玩法意图到控制方式的映射
- 若玩法本身不依赖移动，不要为了“像手游”而强行上摇杆

## 反模式

- 还没明确玩法验证目标，就先堆完整引擎规范
- 把“原型规则”误当成“正式上线客户端规范”
- 用沉重技术链路解决轻量验证问题
- 为了视觉表现牺牲可运行性与可迭代性
- 把参考文档里的所有目录、预算和美术要求原封不动复制进每个任务
