# 渐进式披露规则

`game-agents` 内所有 Agent 必须遵循渐进式披露（Progressive Disclosure）原则，防止上下文膨胀、职责混杂和重复注入。

## 核心原则

1. **默认最小加载**
   - Agent 启动时，只加载本司完成当前任务所必需的最小规则、最小技能和最小上下文。
   - 禁止把全部项目规则、全部审查方法论、全部竞品资料一次性塞入单个任务上下文。

2. **按需上浮**
   - 本司私有信息放在 `agents/<id>/SOUL.md`
   - 项目内多人复用的方法论放在 `skills/` 或 `rules/`
   - 全局通用能力放在 `~/Project/Agents`
   - 只有当任务明确需要时，才逐层向上加载

3. **先创作，后审查**
   - 创作型任务默认先使用本司核心职责与必要参考资料
   - 只有当任务要求“审查 / 验算 / 汇总 / 修复”时，才调用项目共用审查 skill

4. **先摘要，后细节**
   - 输出时优先给结论、框架、关键风险
   - 只有用户要求或流程节点要求时，才展开完整检查清单、公式细节、边界说明

5. **协议按场景加载**
   - `rules/review-report-protocol.md` 仅在输出结构化审查报告时使用
   - 禁止在普通策划创作任务中无差别注入整套审查协议

## 对各类 Agent 的具体要求

- **太子 / 中书 / 尚书**：
  - 优先加载流程、拆解、调度相关规则
  - 仅在需要校验或汇总时加载审查能力

- **门下省**：
  - 默认加载 `review-synthesis`
  - 但不应在每次任务里展开所有子审查清单，除非已经收到了对应维度的问题输入

- **11司**：
  - 先执行本司主职能力
  - 当任务带有“评审、校验、风险识别、返工”性质时，再调用对应审查 skill

## 反模式

- 在 `SOUL.md` 里复制整份长协议
- 在单个 Agent 提示词中同时注入多个不相关审查框架
- 还没明确任务目标，就先加载全套技能与规则
- 把项目共用 skill 的完整示例全文重复写进每个 Agent
