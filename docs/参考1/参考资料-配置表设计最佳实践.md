# 配置表设计最佳实践

本文档提供游戏配置表设计的最佳实践和常见模式。

## 通用设计原则

### 1. 命名规范

**表名：**
- 使用 PascalCase 或 snake_case
- 以 Config 或 Table 结尾
- 示例：`ItemConfig`, `item_config`, `MonsterTable`

**字段名：**
- 使用 snake_case（推荐）或 camelCase
- 清晰描述字段用途
- 避免缩写（除非是行业通用缩写）
- 示例：`item_id`, `unlock_level`, `max_count`

### 2. ID 设计

**唯一标识符：**
- 每个配置项必须有唯一 ID
- ID 通常为整数类型
- 建议使用分段 ID：
  - 1000-1999: 物品类型A
  - 2000-2999: 物品类型B
  - 10000-19999: 任务
  - 20000-29999: 奖励

**ID 字段命名：**
- 主键：`id`, `xxx_id` (如 `item_id`)
- 外键：`xxx_id` (如 `reward_id` 关联 RewardConfig 表)

### 3. 数据类型选择

| 数据类型 | 用途 | 示例 |
|----------|------|------|
| int | 整数、ID、数量、等级 | `item_id: 1001`, `count: 10` |
| float/double | 小数、百分比、系数 | `rate: 0.15`, `multiplier: 1.5` |
| string | 文本、名称、描述 | `name: "新手剑"`, `desc: "攻击+10"` |
| bool | 布尔值、开关 | `is_active: true`, `can_trade: false` |
| JSON string | 复杂结构、数组 | `rewards: [{"type":1,"id":100,"count":5}]` |
| timestamp | 时间点 | `start_time: "2024-01-01 00:00:00"` |
| enum (int) | 枚举类型 | `quality: 1` (1=普通,2=稀有,3=史诗) |

### 4. 常用字段模式

**基础信息字段：**
```
id              int     唯一标识
name            string  名称
description     string  描述
icon            string  图标资源路径
sort_order      int     排序权重
```

**解锁条件字段：**
```
unlock_level    int     解锁等级要求
unlock_quest    int     前置任务ID（0为无）
unlock_vip      int     VIP等级要求（0为无）
unlock_condition string 复杂条件（JSON格式）
```

**时间相关字段：**
```
start_time      string  开始时间
end_time        string  结束时间
duration        int     持续时长（秒）
cooldown        int     冷却时间（秒）
reset_type      int     重置类型（1=每日,2=每周,3=每月）
```

**状态和开关字段：**
```
is_active       bool    是否激活
is_visible      bool    是否可见
status          int     状态（1=正常,2=维护,3=下线）
```

## 常见配置表类型

### 1. 物品配置表 (ItemConfig)

```
item_id         int     物品ID
item_name       string  物品名称

icon_path       string  图标路径
description     string  物品描述
effects         string  效果（JSON）
```

### 2. 任务配置表 (QuestConfig)


### 5. 商店配置表 (ShopConfig)

## JSON 字段设计范例

### 奖励列表 JSON 格式
```json
[
  {"type": 1, "id": 10001, "count": 5},   // 物品
  {"type": 2, "id": 1, "count": 1000},    // 金币
  {"type": 3, "count": 500}                // 经验
]
```

### 解锁条件 JSON 格式
```json
{
  "level": 15,
  "quests": [1001, 1002],
  "items": [{"id": 2001, "count": 10}],
  "vip": 3
}
```

### 技能效果 JSON 格式
```json
{
  "damage": {"base": 100, "scale": 1.5, "type": "physical"},
  "buff": {"type": "attack_up", "value": 20, "duration": 10},
  "aoe": {"range": 5.0, "shape": "circle"}
}
```

### 怪物波次 JSON 格式
```json
[
  {"wave": 1, "monsters": [{"id": 1001, "count": 5, "delay": 0}]},
  {"wave": 2, "monsters": [{"id": 1002, "count": 3, "delay": 10}]},
  {"wave": 3, "monsters": [{"id": 1003, "count": 1, "delay": 20}]}
]
```

## 配置表关系设计

### 一对多关系
- 主表：ChapterConfig (章节配置)
- 从表：StageConfig (关卡配置)
- 关联：StageConfig.chapter_id → ChapterConfig.chapter_id

### 多对多关系（通过中间表）
- 表A：QuestConfig (任务配置)
- 表B：RewardConfig (奖励配置)
- 中间表：QuestRewardMap (任务奖励映射)

### 关联字段命名
- 外键字段应清晰表明关联的表：`reward_table_id`, `drop_table_id`
- 使用 0 或 -1 表示"无关联"或"不存在"

## 常见错误和注意事项

### ❌ 避免的做法

1. **硬编码魔法数字**
   ```
   错误：status: 1, 2, 3 （含义不明）
   正确：在文档中说明或使用枚举
   ```

2. **字段过度复杂**
   ```
   错误：data: "1001:5|1002:10|1003:15" （自定义格式）
   正确：使用 JSON 格式
   ```

3. **缺少默认值说明**
   ```
   错误：没有说明 0 或空字符串的含义
   正确：明确说明 0 表示"无限制"，空字符串表示"无"
   ```

4. **字段名不一致**
   ```
   错误：同一含义在不同表中使用不同字段名
   正确：统一命名规范，如都使用 item_id
   ```

### ✅ 推荐的做法

1. **使用注释和说明**
   - 每个字段都应有清晰的说明
   - 枚举值应列出所有可能的值和含义

2. **考虑扩展性**
   - 预留字段或使用 JSON 存储扩展数据
   - 设计时考虑未来可能的需求

3. **数值平衡考虑**
   - 设计合理的数值范围
   - 考虑数值通胀问题
   - 预留调整空间

4. **版本控制**
   - 配置表应有版本号
   - 记录重要的修改历史

## 配置表示例：完整的每日任务系统

### DailyQuestConfig (每日任务配置)
```
quest_id        int     任务ID
quest_name      string  任务名称
quest_desc      string  任务描述
difficulty      int     难度（1=简单,2=普通,3=困难）
unlock_level    int     解锁等级
target_type     int     目标类型（1=击杀怪物,2=完成关卡,3=消费金币）
target_id       int     目标ID（怪物ID/关卡ID等）
target_count    int     目标数量
reward_id       int     奖励ID（关联 RewardConfig）
activity_bonus  int     活跃度奖励
sort_order      int     显示排序
is_active       bool    是否启用
```

### DailyQuestReward (每日任务奖励)
```
reward_id       int     奖励ID
reward_items    string  物品奖励（JSON）
reward_gold     int     金币奖励
reward_exp      int     经验奖励
```

### DailyActivityReward (活跃度宝箱)
```
box_id          int     宝箱ID
activity_point  int     所需活跃度
reward_items    string  奖励物品（JSON）
is_repeatable   bool    是否可重复领取
```

这样设计的好处：
- 职责清晰：任务、奖励、活跃度分离
- 易于扩展：可以灵活调整任务和奖励
- 可维护性强：修改奖励不影响任务配置
