# 游戏策划多 Agent 协作架构规划（基于三省六部制）

## 1. 核心理念

本项目受 [Edict (三省六部制架构)](https://github.com/cft0808/edict) 启发，结合游戏开发的实际需求，旨在构建一个**制度化、可控、有审核机制**的多 Agent 协作游戏策划系统。

传统的 Multi-Agent 框架（如 AutoGen、CrewAI）往往让 Agent 自由对话，导致最终输出质量不可控且难以回溯。本架构引入了古代帝国的“分权制衡”思想，将工作流切分为：**意图分拣 $\rightarrow$ 方案起草 $\rightarrow$ 质量审核 $\rightarrow$ 派发调度 $\rightarrow$ 专职并行执行**。通过设立“门下省”作为必须经过的审核关卡，确保任何方案在进入细化设计前都符合游戏核心愿景和质量标准。

## 3. 角色与职责定义

### 权限矩阵

为了保证“分权制衡”，各部门之间的任务指派必须遵循严格的权限矩阵（谁能给谁发消息）：

| 发起方 ↓ \ 接收方 → | 太子 | 中书 | 门下 | 尚书 | 11司(六部) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **太子** | — | ✅ | | | |
| **中书省** | ✅ | — | ✅ | ✅ | |
| **门下省** | ✅ | — | ✅ | | |
| **尚书省** | ✅ | ✅ | — | ✅ | ✅ |
| **11 司 (执行层)** | ✅ | | | | — |

*注：此矩阵确保任务必然经过中书省的规划和门下省的审核，尚书省作为唯一的中枢进行具体的向下分发。*

### 任务状态流转

```text
皇上 → 太子分拣 → 中书规划 → 门下审议 → 尚书派发 → 各司执行中 → 待审查 → ✅ 已完成
                      ↑          │                              │
                      └──── 封驳 ─┘                    阻塞 Blocked
```

### 👑 指挥与控制层
- **Boss / 制作人 (人类)**：提出想法、核心目标、商业化要求、美术风格等顶层约束。
- **太子 (主策划 Agent)**：作为统一入口。若是闲聊或简单提问则直接回复；若是具体开发需求，则将其结构化为正式“旨意”向下传递。

### 🏛️ 三省中枢层 (大脑)
- **📜 中书省 (方案起草 Agent)**：游戏的核心系统架构师。负责把 Boss 给出的一句话或一段话需求，扩写成完整的“游戏方案设计大纲”，并明确这个方案需要哪几个“司”来协助细化（拆解子任务）。
- **🔍 门下省 (审议封驳 Agent)**：质量保证 (QA) 与核心体验守护者。拿到中书省的方案后，严格审查是否偏离了 Boss 的核心愿景，是否存在逻辑漏洞或不可实现的功能。如果不达标，直接“封驳”打回中书省重写；如果达标，则“准奏”。
- **📮 尚书省 (派发整合 Agent)**：项目经理 (PM)。拿到通过审核的方案大纲后，将子任务精准派发给下方的 11 个专职策划司。所有司完成工作后，尚书省负责收集、排版、解决设定冲突，最终形成完整的“策划案奏折”上报给制作人。

### ⚔️ 11 司执行层 (专职策划 Agent)
各司为具体的细分领域专家，互不干涉，仅接收尚书省的调度并交付专业内容：
1. **定位司 (Positioning)**：负责受众分析、核心卖点提炼、市场定位。
2. **叙事司 (Narrative)**：负责世界观构建、剧情大纲、角色人设、台词编写。
3. **战斗司 (Combat)**：负责战斗机制、技能设计、受击反馈、敌兵 AI 逻辑。
4. **关卡司 (Level)**：负责场景结构设计、流程引导、怪物与资源分布、解谜设计。
5. **玩法司 (Gameplay)**：负责除战斗外的核心循环（Core Loop）、小游戏玩法设计。
6. **系统司 (Systems)**：负责背包、公会、成就、任务追踪等周边外围系统设计。
7. **UX 司 (User Experience)**：负责交互逻辑、界面层级、UI 表现、玩家心流引导。
8. **数值司 (Balancing)**：负责伤害公式、成长曲线、概率池、属性投放等数学模型构建。
9. **付费司 (Monetization)**：负责商业化模式、抽卡/通行证设计、礼包结构。
10. **经济司 (Economy)**：负责游戏内产出与消耗模型、通货膨胀控制、交易行机制。
11. **运营司 (LiveOps)**：负责开服活动、节日活动排期、留存促活策略设计。

### 🛠️ 通用支撑组件
- **Explore**：相当于本地知识库 RAG，让所有 Agent 能够查阅《游戏世界观手册》或已有的设定，避免前后吃书。
- **Librarian**：具备联网能力，负责查阅市场上如《原神》《黑神话：悟空》等竞品的具体设计细节。
- **Oracle**：独立的高阶逻辑检验器，可供系统司、数值司调用，用于计算复杂公式的极值或验证经济系统是否会崩溃。

### 🔍 审查方法论吸收（参考 `ai_tribunal_design`）

本项目不直接平移一套独立的“AI 合议庭”组织，而是吸收其中写得最好的部分，按渐进式披露拆到现有 11 司与门下省体系中：

1. **保留的精华**
   - 角色身份写法：角色定位、职责边界、不要做什么
   - 审查方法论：检查清单、严重性分级、结构化上报
   - 闭环思想：发现问题 → 汇总 → 修正 → 必要时重审
2. **不直接照搬的部分**
   - 不额外新建一套与 11 司平行的 Agent 组织
   - 不把冗长示例、长篇协议、完整法庭流程硬塞进单个 `SOUL.md`
3. **落地方式**
   - `game-agents/rules/review-report-protocol.md`：统一审查输出协议
   - `game-agents/skills/*.md`：沉淀项目共用审查方法论
   - `game-agents/agents/<id>/SOUL.md`：只保留本司需要调用的审查视角与简短规则

映射原则如下：

- **系统司 / 战斗司**：参考 `system_architect.md` 的结构、边界、依赖与状态机检查
- **叙事司 / 玩法司**：参考 `content_reviewer.md` 的术语、规则一致性与参数完整性检查
- **UX司 / 玩法司 / 关卡司 / 运营司**：参考 `experience_validator.md` 的玩家旅程、心流、认知负荷检查
- **数值司**：参考 `balance_auditor.md` 的公式、成长曲线、概率与 PvP/PvE 强度检查
- **经济司**：参考 `balance_auditor.md` 的经济闭环、投放回收、通胀与套利检查
- **付费司**：参考 `balance_auditor.md` 的付费公平性、保底与概率机制检查
- **门下省**：参考 `orchestrator.md`、`aggregator_judge.md`、`design_fixer.md` 的汇总、保守裁决与有限重审思想

## 4. 工作流实战示例

**场景：制作人下旨：“我们要在一个赛博朋克风的肉鸽（Roguelike）游戏里加一个钓鱼系统，并且和付费抽卡结合。”**

1. **太子**：识别出这是“需求立项”，清洗意图并传达给中书省。
2. **中书省**：起草大纲：“钓鱼作为局外养成系统；使用赛博义体鱼竿；钓到的机械鱼可兑换抽卡资源”。拆解子任务发往门下省审核。
3. **门下省**：审查发现“肉鸽游戏节奏极快，钓鱼可能打断心流”，提出修改意见：“必须限定在安全屋（Hub）区域进行”，如果不改直接**封驳**。
4. **中书省**：修改后重交，门下省**准奏**。
5. **尚书省**：拿到大纲，进行分发：
   - 派发给**玩法司**：设计钓鱼的具体小游戏机制。
   - 派发给**数值司**：计算机械鱼兑换抽卡资源的产出率，防止付费系统崩盘。
   - 派发给**付费司**：设计基于钓鱼的特殊礼包。
   - 派发给**叙事司**：写一段为什么赛博朋克世界还有机械鱼的背景说明。
6. **11 司 (并行执行)**：各司调用 **Explore** 查阅本项目设定，调用 **Librarian** 查询《星露谷物语》钓鱼系统作为参考，生成具体案子。
7. **尚书省**：回收所有案子，整合为一份结构化的《赛博肉鸽-钓鱼系统完整策划案》，上报 Boss。

## 5. 组件与规范三层复用架构 (3-Tier Reuse Architecture)

为了保证扩展性以及避免造轮子，本项目以及下属所有 Agent 必须遵循以下 **“三层复用原则”**：

1. **底层 (Agent 专属)**
   - **存放位置**：`game-agents/agents/<具体部门>/`
   - **内容**：当前 Agent **绝对独占**的 Prompt 模板 (`SOUL.md`)、极特殊逻辑或无法被其他部门调用的专用工具。
2. **中层 (项目共用)**
   - **存放位置**：`game-agents/skills/` 和 `game-agents/rules/`
   - **内容**：当前《游戏策划》项目内，如果某个能力被两个或以上部门同时用到（比如 11 司共用的本地文档检索机制 `Explore`、或共用的某个美术格式规则），**必须提升提取**到本层，供项目内所有 Agent 共享。
3. **顶层 (Agents 中台通用)**
   - **存放位置**：`~/Project/Agents/` (全局中台)
   - **内容**：所有不带具体游戏业务属性的基础能力。如全局通用的 Git 工作流、Markdown 抓取 (`web_fetch`)、代码审查机制、基础 MCP 配置。
   - **约束**：所有新建或新增的 Agent，**必须默认**调用中台能力，严禁在 `game-agents` 中重写中台已有的规则和技能。

## 6. 项目落地与实施计划

1. **第一阶段：中枢与骨架搭建**
   - 在 `game-agents/agents/` 下建立对应的角色目录（如 `taizi/`, `zhongshu/`, `menxia/` 等）。
   - 为每个 Agent 编写 `SOUL.md`（或 Prompt 模板），明确其能力边界和输入输出格式。
2. **第二阶段：打通流转协议**
   - 制定 Agent 之间通信的 JSON/Markdown 格式规范（如 `status: "准奏" | "封驳"`, `feedback: "..."`）。
   - 编写 `scripts/` 中的 Python/Node 脚本模拟流转过程（如 `python scripts/workflow_demo.py`）。
3. **第三阶段：11 司的专职化训练**
   - 针对 11 司分别注入专业的游戏策划知识（如付费司需要懂得 ARPU, LTV, 渗透率等概念）。
   - 接入通用支撑工具（Librarian 的网页搜索功能、Explore 的本地文件读取）。
4. **第四阶段：可视化看板（选做）**
   - 类似 Edict，实现一个简单的网页 Dashboard，实时监控一纸“旨意”流转到了哪个部门、是否被门下省打回、各司的并行进度等。
