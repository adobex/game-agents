# Diagram Output Strategy

用于统一 `game-agents` 中“图表 / 图形化输出”的职责、格式和使用边界。

本文件解决两个问题：

1. 图表不仅给人看，也要能为下游 Agent 提供稳定输入
2. 图表不能变成纯展示层，必须和结构化正文保持同源

## 1. 核心原则

### 1.1 双层交付

所有需要图表化表达的正式产物，默认拆成两层：

- **结构层**：给下游 Agent 消费的结构化事实
- **视图层**：给人做阶段确认的图表/图形化摘要

图表不是唯一信息源，结构层才是单一事实来源。

### 1.2 同源派生

- 图表必须从正文中的结构化信息派生
- 不允许图里出现正文没有定义的关键结论
- 图和正文冲突时，以结构层为准

### 1.3 先结构，后视图

默认输出顺序：

1. 摘要结论
2. 结构层正文
3. 图表视图
4. 风险 / 待决项
5. 下一步流转

### 1.4 渐进式披露

- 不是每个任务都必须上重图
- 先判断图是否能降低理解成本，再决定是否输出
- 若本轮问题只需局部确认，可只输出一个最关键视图

## 2. 双层交付契约

当某个 Agent 适合做图时，建议遵循如下骨架：

```markdown
## 摘要结论
- [一句话判断]

## 结构层
- [节点/模块/阶段/关系/风险的结构化说明]
- [必要时用 Markdown 表格]

## 视图层
### 图 1：[图名]
- 图类型：Mermaid / HTML Chart / Table View
- 用途：给人快速确认什么
- 对应结构层：[# 某小节]

```mermaid
...
```

## 风险与待决项
- [...]

## 下一步流转
- [...]
```

## 3. 技术栈选择

### 3.1 Mermaid

默认用于：

- 架构图
- 流程图
- 状态机图
- 依赖关系图
- 页面流 / 导航图

优点：

- 文本化，方便版本管理
- 适合下游 Agent 直接读取源码
- 适合随方案迭代

限制：

- 精细视觉控制有限
- 不适合复杂汇报型信息图

### 3.2 轻流程条 / 轻关系视图

默认用于：

- Core Loop 快速确认
- 页面主路径概览
- 阶段结构的一屏展示

说明：

- 适合 HTML / GDD 页面里给人快速扫读
- 可以比 Mermaid 更轻，但仍应由结构层派生
- 只适合作为“主路径提示”，不代替正式结构说明

### 3.3 Markdown 表格

默认用于：

- 节奏表
- 对照表
- 风险矩阵底表
- 节点清单
- 依赖表

说明：

- 表格不是“低配图”，而是最稳的结构层表达
- 若图表存在，通常仍应保留对应表格或字段块

### 3.4 HTML / ECharts

默认用于：

- `analyst` 的发行复盘 / 演示型报告
- 面向人做阶段确认的高密度图表页
- 页面级总结、Hero 数据卡、图表区块、结论收束

说明：

- 只适合“报告型”交付
- 不应替代结构层正文
- 默认仍要保留可供下游消费的结构化文字与表格
- 若是文档型 HTML，ECharts 默认优先 `renderer: 'svg'`

### 3.5 不建议

- 只交一张纯图片，不给结构化说明
- 只给 HTML 图，不给对应文字结论
- 为了“显得高级”给所有 Agent 都强行补重图

## 5. 推荐图型与边界

### `systems`

- 推荐图：模块图、状态机图、跨系统依赖图
- 不建议：只画页面草图代替系统边界说明

### `ux`

- 推荐图：页面流、导航层级图、关键任务路径图
- 不建议：只给视觉描述，不给路径和层级

### `shangshu`

- 推荐图：任务依赖图、串并行流转图、回收时序图
- 不建议：只列任务清单，不说明依赖与前后顺序

### `menxia`

- 推荐图：风险矩阵、争议归并图、封驳决策树
- 不建议：为了好看把裁决逻辑藏进图里，正文反而缺失

### `positioning`

- 推荐图：竞品坐标图、受众-卖点矩阵、差异化对照图
- 不建议：只做大而空的象限图，不落到用户和卖点

### `liveops`

- 推荐图：活动日历、节奏图、生命周期节点图
- 不建议：只有时间线，没有活动目标和指标假设

### `monetization`

- 推荐图：付费路径图、商品矩阵、价格带分层
- 不建议：只画漏斗，不补公平性和经济约束说明

### `analyst`

- 推荐图：渠道占比、投放节奏、素材矩阵、竞品雷达
- 不建议：只有图表页，没有证据等级和下游转译

## 6. 何时必须出图

满足以下任一条件时，建议视为“必须至少补一张图”：

- 任务核心是 **结构关系**
- 任务核心是 **流程 / 路径**
- 任务核心是 **时间节奏**
- 任务核心是 **多对象对照**
- 该产物需要给人做阶段确认，并且纯文字不利于快速判断

## 7. 下游消费约束

为了让下游 Agent 真正可用，图表输出要满足：

- 每张图都要对应正文中的结构层小节
- 图中的节点名、阶段名、模块名要与正文一致
- 若使用 Mermaid，优先用清晰节点名，不用过度缩写
- 若使用轻流程条，节点顺序和命名必须能回指到正文阶段定义
- 若使用 HTML 图，正文中仍要保留可引用的结论和表格
