# Feature Design Template

用于把某个具体功能写成接近可研发协作的设计文档。相比 `docs/output-templates.md` 中的专项方案模板，这份模板更适合 `gameplay`、`systems`、`combat`、`economy`、`monetization` 等司输出可交付给研发、测试、数值继续展开的功能规格。

## 何时使用

- 需要把专项方案进一步写成“功能设计文档”时
- 需要明确客户端 / 服务器职责边界时
- 需要同步配置表设计、交互协议、边界情况时

## 写作原则

- 先写目标与核心循环，再写实现侧逻辑
- 重点描述规则、数据流、职责边界，不堆 UI 像素细节
- 所有可能频繁调整的数值尽量配置化
- 对异常、断线、资源不足、并发冲突至少给出最小处理策略

```markdown
- 为玩家提供什么价值

## 2. 核心玩法概述

### 2.1 入口与退出
- 入口位置：
- 入口条件：
- 触发方式：
- 退出方式：
- 退出后去向：

### 2.2 核心循环
- 开始阶段：
- 玩家主要操作：
- 系统响应：
- 状态变化：
- 结束条件：
- 奖励与结算：

## 3. 客户端与服务器逻辑

### 3.1 客户端逻辑
- 显示与交互：
- 本地校验：
- 数据缓存：
- 表现层反馈：

### 3.2 服务器逻辑
- 数据持久化：
- 业务校验：

- 数据同步：

### 3.3 关键交互协议
- 协议 1：进入功能
- 协议 2：执行核心操作
- 协议 3：领取 / 结算

## 4. 配置表设计
- 表名：

- 高风险点：

## 7. 下一步流转
- 交给谁继续展开：
```

## 最小通过标准

- 能回答“玩家怎么进、怎么玩、怎么结束”
- 能回答“客户端做什么、服务器做什么”
- 至少列出 1-2 个关键交互协议
- 至少列出 1 张配置表
- 至少覆盖 2 个边界情况
