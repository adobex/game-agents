# GDD HTML Patterns

用于沉淀 `docs/参考2/game-design-doc_eb5649ca.html` 中值得被 `game-agents` 吸收的页面结构、信息表达与图表实现方式。

本文件不要求机械复制参考页的题材包装，而是提炼其对本项目长期有价值的模式。

## 1. 适用范围

优先适用于：

- `zhongshu` 的《宏观规划大纲》HTML 版
- `gameplay` 的《玩法方案》HTML 版
- `systems` 的《系统设计方案》HTML 版
- `ux` 的《交互与界面方案》HTML 版
- `positioning` 的《定位报告》HTML 版
- `shangshu` 的《派工单》/《整合奏折》展示版
- `menxia` 的《审议结论》展示版
- `liveops` 的《运营排期方案》HTML 版
- `monetization` 的《商业化方案》HTML 版
- `balancing` 的《数值模型方案》HTML 版
- `economy` 的《经济循环方案》HTML 版
- 需要给人做阶段确认的 GDD / 设计大纲 / 实施说明页

不直接适用于：

- 纯审查型短报告
- 仅供下游 Agent 消费的最小结构稿
- 必须极轻量返回的一次性问答

## 2. 页面级信息架构

参考页最值得吸收的，不是某个视觉皮肤，而是**连续阅读式信息架构**：

1. `Hero`
2. `目录 / 章节导航`
3. `项目概述`
4. `核心循环 / 主路径`
5. `按系统或专题分章节展开`
6. `技术架构 / 数据结构 / 存档结构`
7. `结尾收束`

对 `game-agents` 的建议：

- 正式 HTML 页优先按“总览 -> 主循环 -> 分系统 -> 技术/数据结构”的顺序组织
- 不是一开始就进入参数和细项
- 让阅读者先建立全局模型，再看局部设计

## 3. 章节表达模板

单个章节建议采用：

1. 章节标题
2. 一句话导语
3. 关键判断或核心承诺
4. 结构化正文
5. 图表 / 视图层
6. 必要时补参数表或数据结构

说明：

- 图表应服务该章节的核心判断
- 章节开头最好先告诉读者“这一章想说明什么”
- 不要只堆内容块而没有主线

## 4. 值得吸收的表达手法

### 4.1 项目概述双栏

参考页把“定位”和“技术参数”并排展示，很适合快速建立项目心智。

建议吸收为：

- 左栏：体验定位 / 用户价值 / 题材 / 目标
- 右栏：平台 / 技术边界 / 分辨率 / 运行约束 / 关键模块

### 4.2 核心体验承诺卡片

参考页用 3 张卡片提炼“爽快战斗 / 构筑乐趣 / 西游探索”，很适合在 HTML 页头部确认本轮主卖点。

建议吸收为：

- 每份 GDD 型页面最多 3-4 张核心承诺卡
- 只放最重要的体验目标，不把所有 feature 都做成卡片

### 4.3 轻流程条

参考页的核心循环没有强行上重流程图，而是用一排节点 + 箭头表达主循环。

适用场景：

- 给人快速确认主流程
- 还没必要画正式 Mermaid
- 需要比纯文字更容易扫读

建议：

- 视图层允许使用“轻流程条”作为 Mermaid 的补充
- 但正文里仍应保留结构化主循环说明

### 4.4 关键结构直接可见

参考页直接展示：

- 场景流程表
- 技术目录树
- 存档结构

这非常值得吸收，因为它让“系统真实结构”而不是“概念描述”直接暴露出来。

建议：

- `systems`：直接展示模块树、状态字段、存档结构
- `gameplay`：直接展示规则结构、房间类型、奖励层级
- `zhongshu`：直接展示系统树与司局分工

## 5. 图文表的角色分工

参考页最值得吸收的一点是：**图、文、表各干各的活**。

建议在本项目中固定分工：

- **正文**：给出判断、理由、边界
- **表格**：给出结构层和可供下游消费的事实
- **图表**：帮助人快速确认趋势、分布、差异和关系

禁止：

- 只有图，没有正文判断
- 只有展示卡片，没有结构化信息
- 把图表当成唯一事实源

## 6. 图表策略

### 6.1 什么时候用 ECharts

适合：

- 属性对比
- 分布占比
- 节奏变化
- 对象对照
- 稀有度 / 权重 / 分层

不适合：

- 非要把所有结构关系都图表化
- 本可用表格表达却硬上图

### 6.2 优先使用 SVG renderer

参考页使用：

```js
const chart = echarts.init(dom, null, { renderer: 'svg' });
```

建议本项目默认遵循：

- GDD / 文档型 HTML 图表优先用 SVG renderer
- 这样更适合缩放、导出、静态归档和文档嵌入

### 6.3 图表区块要成组出现

参考页的图表通常作为章节收束，而不是零碎乱插。

建议：

- 每章最多 1-2 个关键图表
- 图表前先有文字说明
- 图表后可补一行解释其对应的判断

## 7. 视觉层面值得学什么

### 值得吸收

- Hero + 阅读进度条
- 清晰的章节标题层级
- 统一的卡片体系
- 表格有明显的表头 / hover / 视觉分区
- 图表、表格、代码块都有自己的容器

### 不建议直接复制

- 题材化皮肤本身
- 大量 inline style
- 为了“成品感”牺牲后续模板维护性

## 8. 对各 Agent 的落地建议

### `zhongshu`

重点吸收：

- 概述双栏
- 核心体验承诺卡
- 系统树与分工可见
- 主循环 / 阶段结构先行

### `gameplay`

重点吸收：

- 轻流程条
- 房间 / 节点 / 规则类型的结构化可视化
- 升级、构筑、变体的图文表组合

### `systems`

重点吸收：

- 技术架构目录树
- 状态 / 场景流转表
- 存档与数据结构直接展示

### `ux`

重点吸收：

- 页面层级与关键路径前置
- 高频任务的轻流程条
- 单屏职责和导航关系成组展示

### `positioning`

重点吸收：

- 概述双栏
- 用户画像 / 竞品 / USP 的图文表组合
- 差异矩阵或定位视图服务阶段确认

### `shangshu`

重点吸收：

- 总览卡片先讲本轮目标、并行范围与回收标准
- 依赖链和串并行关系显性化
- 整合稿里先给收束结论，再展示冲突归并关系

### `menxia`

重点吸收：

- 裁决先行，理由和阻塞项紧随其后
- 风险等级、争议项和优先级成组展示
- 让人一眼看懂“为什么准奏 / 为什么封驳”

### `liveops`

重点吸收：

- 版本目标与生命周期阶段先行
- 活动日历、节点密度和空窗期显性化
- KPI 假设与活动目标绑定展示

### `monetization`

重点吸收：

- 商业化框架与用户分层先行
- 付费路径、商品矩阵和价格带成组展示
- 公平性约束与收益假设同时可见

### `balancing`

重点吸收：

- 关键公式、关键等级节点和验算结论前置
- 成长曲线、概率梯度和节点对比成组展示
- 边界条件与失衡预警直接可见

### `economy`

重点吸收：

- 产出-库存-消耗闭环先行
- 货币层级、兑换关系和沉淀池成组展示
- 通胀、套利与长线健康预警直接显性化

**吸收它的页面级组织方式、图文表分工和关键结构显性化方法，不吸收它的一次性成品写法和重内联实现。**
